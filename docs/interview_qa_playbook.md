# Multi-Agent Business Analysis 面试问答手册

本手册基于当前项目实现，按实习/秋招高频面试路径定制。  
建议使用方法：
- 先背 `30 秒项目介绍` + `3 分钟深挖版`
- 再按岗位方向重点准备对应题组（后端/AI 应用/工程化）
- 每个回答尽量带 1-2 个可量化指标（耗时、成功率、重试次数、token）

---

## 0. 开场自我介绍模板

### Q0-1: 用 30 秒介绍你的项目
**高分回答：**  
我做了一个基于 `LangGraph + FastAPI` 的多智能体企业尽调系统。用户输入公司信息后，系统先生成分析师角色方案，在 `human_feedback` 节点中断等待人工确认；用户可多轮反馈重生分析师，满意后再进入并行访谈检索，最终自动汇总报告并导出 `DOCX/PDF`。  
工程上我实现了一个轻量任务运行时，支持任务状态机、事件日志、失败自动重试和手动重试，同时采集 token 与时延指标，保证流程可观测、可恢复。

### Q0-2: 用 3 分钟展开讲这个项目
**高分回答：**  
这个项目的核心目标是把“LLM demo”做成“可执行业务流程”。我把系统拆成三层：  
1) **API 层（FastAPI）**：处理登录、任务创建、反馈提交、进度查询、下载报告。  
2) **任务运行时层（TaskRuntime）**：将长耗时流程异步化，维护 `pending -> running_generation -> awaiting_feedback -> running_feedback -> completed/failed` 状态流转，并持久化任务和事件。  
3) **智能体编排层（LangGraph）**：主图负责任务全局编排，子图负责单分析师访谈链路（提问、搜索、回答、写章节）。  

在体验上，我重点做了 **Human-in-the-loop**：不是一次生成结果，而是让用户在分析师方案阶段可多轮反馈迭代，这样业务可控性更高，也更贴近真实工作流程。  
在工程上，我做了失败恢复（自动重试 + 手动重试 + 重启后恢复中断任务）、运行指标统计（LLM 调用次数、token、耗时）和导出能力（docx/pdf），让系统具备交付属性。  
如果要进一步生产化，我会优先升级会话管理、任务存储与队列系统，并补充测试与压测体系。

## 1. 项目动机与需求分析

### Q1-1: 你为什么做这个题目？
**高分回答：**  
我想解决两个痛点：  
1) 商业尽调信息分散、人工汇总成本高；  
2) 传统 AI 报告“一次性生成不可控”，很难贴合用户关注点。  
所以我把流程设计成“机器生成 + 人工确认 + 机器继续执行”的闭环，在效率和可控性之间找平衡。

### Q1-2: 为什么要多智能体，不用单 Agent？
**高分回答：**  
单 Agent 更像串行思考，容易在复杂议题上覆盖不足。多智能体可以按角色分工并行探索不同维度（如商业模式、规模成长、风险），然后统一汇总，覆盖度和可解释性更好。  
另外，从工程角度，多角色天然适合并发调度，后续也更容易做角色级别的质量评估和替换。

### Q1-3: 你的核心用户价值是什么？
**高分回答：**  
核心价值有三点：  
1) **节省调研时间**：把“搜集-访谈-汇总-成文”自动化。  
2) **提高可控性**：用户可以中途反馈修正分析师方案。  
3) **可交付**：直接导出报告文档，能进业务流程而不只是聊天结果。

## 2. 架构设计与技术选型

### Q2-1: 为什么选 LangGraph，而不是自己写流程代码？
**高分回答：**  
因为这个场景有明确状态机和中断恢复需求。LangGraph 天然支持节点编排、条件路由、并行分支和 checkpoint，更适合构建可控流程，而不是一次性 prompt。  
我自己只保留业务运行时（任务状态、事件、重试）在 FastAPI 侧，形成“编排层 + 运行时层”分工。

### Q2-2: 主图和子图怎么划分的？
**高分回答：**  
主图负责流程级节点：`create_analyst -> human_feedback -> interviews -> write/finalize`。  
子图负责单 analyst 的访谈闭环：`ask_question -> search_web -> generate_answer -> save_interview -> write_section`。  
这样拆分后，主图管“流程控制”，子图管“角色执行逻辑”，可维护性更好。

### Q2-3: 为什么 FastAPI + 模板页面，而不是前后端分离？
**高分回答：**  
项目目标是快速验证业务闭环，FastAPI + Jinja2 上手快、部署简单、调试链路短。  
如果进入产品化阶段，我会迁移到前后端分离：后端保留任务 API，前端用 React/Vue 做更完整的任务中心和实时事件展示。

## 3. 工作流细节深挖

### Q3-1: 人工反馈节点是怎么工作的？
**高分回答：**  
在主图里我把 `human_feedback` 设为 interrupt 点。首次运行到这个节点后暂停，返回分析师预览。用户提交反馈时，我通过 `update_state` 把反馈写回图状态，然后继续 `stream`。  
如果反馈非空，路由到 `regenerate_analyst` 后再次回到 `human_feedback`；若反馈为空，表示确认通过，进入访谈和报告生成。

### Q3-2: 并行访谈怎么实现？
**高分回答：**  
我在 `start_interviews` 后通过 `Send` 给每个 analyst 派发 `conduct_interview` 任务，每个 analyst 都跑一套子图，再在报告写作节点汇总。  
这种方式把“角色并行”和“结果聚合”自然表达出来，便于后续做并发控制和资源配额。

### Q3-3: 你怎么控制访谈轮次和成本？
**高分回答：**  
子图里维护 `turn_count`，通过条件边判断是否继续提问，超过 `max_num_turns` 就收敛到 `save_interview`。  
成本层面通过 token 聚合和节点级 metrics 统计观察热点节点，后续可用来做 prompt 压缩和模型分级调用。

## 4. 工程化与可靠性

### Q4-1: 长任务怎么做异步？怎么避免接口阻塞？
**高分回答：**  
创建任务后立即返回任务页，后台由 `TaskRuntime.run_in_background` 用线程执行真实工作，接口只负责读写任务状态。  
这样用户可通过 `/tasks/{id}` 和 `/tasks/{id}/events` 轮询进度，不会阻塞请求线程。

### Q4-2: 失败恢复如何设计？
**高分回答：**  
我做了三层恢复：  
1) **阶段内自动重试**（每阶段有限次）；  
2) **用户手动重试**（按失败阶段恢复 generation 或 feedback）；  
3) **服务重启恢复**（把中断中的任务标记 failed，允许用户重试）。  
这样至少能把“永久卡死”问题转为“可见、可恢复”问题。

### Q4-3: 你做了哪些可观测性？
**高分回答：**  
任务维度：状态、失败阶段、错误信息、重试次数。  
事件维度：task created/started/completed/failed/retrying 等。  
LLM 维度：节点级 token 和 latency 聚合。  
这些信息既服务 UI 进度展示，也支持后续排障和性能优化。

### Q4-4: 你认为当前实现最大的工程风险是什么？
**高分回答：**  
当前是单机 demo 方案：任务持久化到本地 JSON、会话在内存，扩展到多实例会有一致性与恢复问题。  
如果进生产，我会替换为 Redis + Postgres + 队列（Celery/RQ），并加幂等键、分布式锁和任务去重。

## 5. 安全与数据合规

### Q5-1: 这个项目有哪些安全风险？你会怎么改？
**高分回答：**  
风险主要在会话管理和文件访问：  
1) 内存会话不可持久、可预测；  
2) 下载接口需要严格绑定 `task_id` 与用户；  
3) 密钥管理不能落库或日志泄露。  
改造方案：使用安全随机 session/JWT、HttpOnly + SameSite cookie、统一鉴权中间件、敏感字段脱敏日志、密钥走环境变量与密管系统。

### Q5-2: 你如何避免“模型胡说”导致报告失真？
**高分回答：**  
我采用“检索上下文 + 结构化流程 + 人工确认”三重约束：  
先检索再回答，避免纯生成；关键阶段插入人工反馈；最后可在报告中附来源段落，提升可追溯性。  
下一步我会加入引用一致性校验与事实核对规则。

## 6. 性能优化与成本控制

### Q6-1: 如果响应太慢，你会怎么优化？
**高分回答：**  
我会按“先观测后优化”做三步：  
1) 基于节点 latency 找热点（如 `write_report`、`generate_answer`）；  
2) 优化 prompt 长度、限制上下文窗口、减少无效轮次；  
3) 模型分级（轻模型做检索/改写，强模型做最终整合）。  
最后通过 A/B 评估耗时与质量变化，避免只追求快导致质量下降。

### Q6-2: 成本怎么管？
**高分回答：**  
项目内已聚合 token 与调用次数。我会进一步设置预算阈值和降级策略：  
超预算时减少 analyst 数量、减少轮次、切换性价比模型；并在任务详情显示预计/实际成本，方便产品策略决策。

## 7. 测试与质量保障

### Q7-1: 你会怎么测试这个系统？
**高分回答：**  
我会分三层：  
1) **单元测试**：状态流转、重试逻辑、权限判断；  
2) **集成测试**：从创建任务到完成导出的完整 API 链路；  
3) **回归测试**：固定输入下检查报告结构完整性（章节、风险摘要、推荐段）。  
LLM 结果不做逐字断言，而是做结构化和关键字段断言。

### Q7-2: 如何验证你做的优化有效？
**高分回答：**  
定义基线指标：任务成功率、P50/P95 耗时、平均 token、失败重试后成功率。  
每次优化前后对同一批样本跑对比，确认质量不退化后再合入。

## 12. 项目真实踩坑与修复（可直接当面试案例）

### Case 12-1: 无反馈时 interview 报 `KeyError: 'analyst'`
**现象：**  
用户在 `awaiting_feedback` 阶段不填反馈直接继续，流程进入 `conduct_interview` 时抛错：子图状态缺少 `analyst`。  

**根因：**  
无反馈分支直接路由到 `conduct_interview`，跳过了 `Send(...)` 分发步骤，导致子图拿到的是主图状态而不是 interview 子状态。  

**修复：**  
引入 `start_interviews` 节点，让“有反馈/无反馈”两条路径都先进入 interview 分发；确保子图入参统一包含 `analyst/messages/context/...`。  

**结果：**  
消除 `KeyError`，无反馈路径可稳定继续执行。

---

### Case 12-2: 需要支持“反馈直到满意”为止
**现象：**  
原流程只支持一次反馈，用户无法多轮修正分析师方案。  

**修复：**  
将主图改为循环：`human_feedback -> regenerate_analyst -> human_feedback`（反馈非空时回环）；反馈为空才进入 interview。  
同时在任务状态中增加 `analyst_version`，前端展示 `v1/v2/v3...`，每次重生自动递增。  

**结果：**  
用户可多轮迭代分析师，满意后再生成最终报告，交互更符合业务真实流程。

---

### Case 12-3: 并行分支触发 `max_num_turns` 并发写冲突
**现象：**  
`running_feedback` 阶段报 `INVALID_CONCURRENT_GRAPH_UPDATE`，提示 `At key 'max_num_turns': can receive only one value per step`。  

**根因：**  
并行 interview 分支在同一步回写同一标量键，LangGraph 的默认 last-value 通道不允许并发多写。  

**修复：**  
1) 在 `ResearchGraphState.max_num_turns` 上增加 reducer（`Annotated[int, keep_latest]`）；  
2) 结构上减少回写来源：并行分发时不再重复注入/回写该字段。  

**结果：**  
重试后并发更新冲突消失，反馈阶段稳定性提升。

---

### Case 12-4: 最终报告成功但页面仍显示大段错误
**现象：**  
页面出现“上面报错、下面 Final Report Ready”的冲突展示。  

**根因：**  
任务成功完成后，历史 `error` 字段未清空，前端仍按错误字段渲染。  

**修复：**  
后台任务成功落盘时强制清空 `error/failed_stage`；前端在 `status == completed` 时不展示错误块。  

**结果：**  
UI 状态一致，不再误导用户。

---

### Case 12-5: `search_web` 中 Tavily 返回类型不一致
**现象：**  
报错 `AttributeError: 'str' object has no attribute 'get'`。  

**根因：**  
代码默认把检索结果当 `dict` 处理，但某些模式下 Tavily 返回 `str`。  

**修复：**  
增加类型兼容：`dict` 按 `url/content` 读取，非 `dict` 按纯文本处理并设置默认 `href="#"`。  

**结果：**  
检索阶段鲁棒性提升，不再因返回形态差异导致任务失败。

---

### Case 12-6: Token usage 显示为 0
**现象：**  
任务有耗时和调用，但 token 显示 `0`。  

**根因：**  
1) 部分节点历史上把 token 指标写死为 0；  
2) 不同 provider/兼容层 usage 字段命名不一致（`prompt_tokens` vs `input_tokens`），导致读取不到。  

**修复：**  
1) 改为尽量从响应元数据读取真实 usage；  
2) 扩展字段兼容（`usage_metadata`、`response_metadata.token_usage`、`input/output/prompt/completion/total`）；  
3) 聚合层增加 `usage_available`，前端对不可用显示 `N/A`（而非 0）。  

**结果：**  
指标展示更真实，且不会因 usage 缺失触发失败/重试，避免无谓 token 消耗。

---

### Case 12-7: 提交空反馈时报 `Field required`（422）
**现象：**  
在 `awaiting_feedback` 阶段点击提交但未填写反馈时，接口返回：
`{"detail":[{"type":"missing","loc":["body","feedback"],"msg":"Field required","input":null}]}`，流程无法继续。

**根因：**  
后端 `submit_feedback` 路由把 `feedback` 声明为必填表单字段（`Form(...)`）。  
当请求体里缺失 `feedback` 时，FastAPI 会在参数校验阶段直接返回 422，业务逻辑拿不到请求，无法进入“无反馈也继续”分支。

**修复：**  
1) 将 `feedback` 改为可选输入（`Form("")`），避免参数层直接拦截；  
2) 在业务层做归一化：`feedback.strip()` 为空时注入默认指令  
`"No additional feedback. Please proceed to generate the final report based on the current draft."`；  
3) 统一使用归一化后的 `feedback` 更新任务并继续 `running_feedback`。

**结果：**  
空反馈场景不再报 422，用户可直接继续生成最终报告；同时仍保留“有反馈时按用户意见迭代”的能力。

---

### Case 12-8: `running_feedback` 阶段被热重载打断，页面一直停在 “Applying feedback...”
**现象：**  
用户提交反馈后，任务进入 `running_feedback`，但页面长时间停留在 “Applying feedback...”。  
后台日志出现 `cannot schedule new futures after shutdown`，任务既没有写成 `completed`，也没有写成 `failed`。

**根因：**  
任务在后台线程里继续执行 LangGraph 时，开发环境的 `uvicorn --reload` 因代码变更触发了服务重启。  
进程退出会关闭线程池，但尚未完成的图执行还在提交异步任务，于是抛出 `cannot schedule new futures after shutdown`。  
由于中断发生在业务流程中间，旧逻辑没有把任务状态落成终态，导致前端轮询时一直看到 `running_feedback`，表现成“卡住”。

**修复：**  
1) 在服务启动时扫描任务存储；  
2) 对停留在 `running_generation/running_feedback` 的任务自动标记为 `failed`；  
3) 写入明确错误：`Task interrupted by server reload/restart. Please retry.`；  
4) 追加 `task.interrupted` 事件，便于排查是业务失败还是进程生命周期中断。

**结果：**  
即使再次遇到开发环境热重载中断，任务也不会永久卡在运行中；用户刷新页面后能看到明确失败原因，并可直接点击 `Retry` 重新执行。

---

### Case 12-9: 中断后重试会增加 token 成本，但通常小于整任务重跑
**现象：**  
任务在 `running_feedback` 或报告生成后半段被中断后，用户会担心点击 `Retry` 是否等于整条链路从头再跑一遍，导致 token 成本明显放大。

**根因：**  
LLM 调用一旦发生，已经消耗的 token 不会因为任务中断而“退回”。  
因此重试本质上是一次新的模型调用，确实会产生额外 token；但如果中断点已经接近流程后半段，通常只会重跑失败后的那部分，而不是所有前序步骤。

**修复：**  
我在文档和交互说明里明确向用户解释两点：  
1) 重试一定会增加 token 成本；  
2) 但增量成本通常低于“整条任务完全重做”。  
同时建议结合 `Total Tokens` 和节点级指标观察究竟是哪些阶段被重复执行，为后续做更细粒度恢复和结果缓存提供依据。

**结果：**  
用户对失败重试的成本预期更清晰，也为后续优化方向提供了依据：优先做“失败阶段最小重跑”和“中间结果缓存”，把中断带来的 token 浪费控制在可接受范围内。

## 14. Agent 开发岗技术深挖（定制版）

### Q14-1: 你如何定义一个“合格”的 Agent 系统？
**高分回答：**  
我会用四个维度定义：  
1) **任务完成率**：是否稳定完成目标；  
2) **过程可控性**：是否可中断、可回滚、可人工介入；  
3) **可观测性**：是否有步骤级日志、状态、成本指标；  
4) **可扩展性**：新工具/新角色能否低成本接入。  
在我这个项目里，LangGraph 负责过程可控，TaskRuntime 负责运行可观测，二者配合让 Agent 从“会回答”变成“可运行系统”。

### Q14-2: ReAct、Plan-and-Execute、Graph 编排你怎么选？
**高分回答：**  
我按任务形态选：  
- **ReAct** 适合短链探索，灵活但不可控性高；  
- **Plan-and-Execute** 适合中等复杂任务，能先规划后执行；  
- **Graph 编排** 适合多阶段、可恢复、需人工介入的业务流程。  
我的项目有明显状态机和中断恢复需求，所以优先 Graph；在子图内部保留 ReAct 式检索问答能力。

### Q14-3: Agent 的状态（State）应该怎么设计？
**高分回答：**  
状态设计要满足“最小完备 + 可恢复”。我会把状态分三类：  
1) **业务状态**：company/focus/role 等；  
2) **流程状态**：当前阶段、失败阶段、重试次数；  
3) **模型状态**：messages、context、llm_metrics。  
并确保状态可序列化、可版本化，避免后续 schema 演进时状态不可读。

### Q14-4: 你如何处理上下文爆炸（context window pressure）？
**高分回答：**  
我会做三层控制：  
1) **输入裁剪**：只保留与当前节点目标相关上下文；  
2) **中间摘要**：把访谈历史做结构化摘要，而不是全量拼接；  
3) **分层模型**：便宜模型做摘要/检索改写，贵模型做最终综合。  
目标是把 token 花在“决策关键路径”上，而不是机械搬运历史内容。

### Q14-5: Tool 调用失败或返回异常格式时怎么办？
**高分回答：**  
策略是“防御式工具封装 + 可恢复执行”：  
1) 工具层做输入输出 schema 校验；  
2) 对常见不规范返回（dict/str/空结果）做兼容转换；  
3) 节点失败可重试并记录失败上下文；  
4) 达到阈值后降级到“无工具”路径并标注置信度。  
我在 Tavily 返回类型兼容问题上就用过这套方式，避免流程因工具抖动直接中断。

### Q14-6: 你如何做 Agent 结果评估（Evaluation）？
**高分回答：**  
我会把评估拆成在线和离线：  
- **在线指标**：成功率、耗时、token、重试率；  
- **离线质量**：结构完整性、来源可追溯性、关键信息覆盖率、人工打分。  
对于生成任务，我倾向“规则 + LLM Judge + 人审抽样”三结合，而不是只看主观感受。

### Q14-7: 如何防止 Agent 进入死循环或无效迭代？
**高分回答：**  
我会加硬约束：最大轮次、最大工具调用次数、预算阈值、重复动作检测。  
一旦触发阈值，走明确降级路径（结束并输出未完成原因，或请求人工介入）。  
子图里的 `max_num_turns` 就是最基础的防循环机制，后续可叠加重复语义检测。

### Q14-8: 多 Agent 并行时，如何保证结果一致性？
**高分回答：**  
一致性分为两层：  
1) **数据一致性**：每个分支只写自己的局部结果，汇总节点统一合并；  
2) **语义一致性**：汇总前先做去重和冲突检测，再由主模型统一裁决。  
工程上要避免多个分支直接写同一状态字段，尽量采用 append-only 再聚合。

### Q14-9: Agent 系统如何做幂等与重放（replay）？
**高分回答：**  
关键是“步骤 ID + 输入快照 + 输出快照”。  
对外部副作用操作（写文件、发请求）绑定幂等键；重试时先查是否已成功，避免重复执行。  
在我的项目里，任务级 retry 已有，但如果上生产我会进一步做步骤级幂等和可重放调试。

### Q14-10: 你怎么做 Prompt 工程，避免变成“魔法字符串”？
**高分回答：**  
我会把 prompt 产品化：  
1) 按职责拆模板（分析师生成、搜索改写、报告写作）；  
2) 版本化管理（prompt v1/v2 + 变更说明）；  
3) 用固定评测集做回归，避免“改 prompt 破坏旧能力”。  
本项目里 prompt 已集中在 `prompt_lib`，这是后续做版本治理的基础。

### Q14-11: 你如何设计 Agent 的错误分类？
**高分回答：**  
我会分三层：  
1) **可重试错误**：网络抖动、限流、临时超时；  
2) **不可重试错误**：参数非法、权限失败、状态缺失；  
3) **质量错误**：流程成功但结果不达标。  
不同类型对应不同策略：自动重试、直接失败、或转人工审阅。这样能避免“所有错误都重试”造成成本浪费。

### Q14-12: 为什么你强调 Human-in-the-loop，而不是追求全自动？
**高分回答：**  
在高价值任务里，完全自动往往不现实。真正可落地的是“关键节点让人可控，非关键环节自动化”。  
Human-in-the-loop 不是妥协，而是工程策略：它能显著降低错误传播，提升业务信任，并给系统提供高质量反馈数据。

### Q14-13: 如果让你把这个项目上线，你的 30/60/90 天计划？
**高分回答：**  
**30 天**：补测试与监控，梳理 SLO（成功率、P95、成本）。  
**60 天**：迁移运行时基础设施（Redis + Postgres + 队列），完善鉴权与审计。  
**90 天**：建立评估闭环（离线数据集 + 在线指标看板 + prompt/model A/B），形成持续优化机制。

### Q14-14: 面试官问“你和纯 LLM 调用工程师的区别是什么”怎么答？
**高分回答：**  
我的优势不只在调用模型，而在“Agent 系统工程化”：  
我会把任务拆成可执行图、定义状态机、设计失败恢复、建设可观测指标，并给出生产迁移路径。  
也就是说，我做的是“让能力稳定落地”的工程，不只是“让模型答对一次”。

## 15. 大模型原理 + 编排取舍追问（Agent 岗高频）

### Q15-1: Transformer 为什么有效？和 RNN 比本质区别是什么？
**高分回答：**  
核心是自注意力能直接建模任意位置 token 的关系，且并行计算效率高。  
RNN 强在序列递归，但长依赖难学、训练难并行；Transformer 通过多头注意力 + 残差 + 归一化，在规模化训练和泛化上更占优。  
对 Agent 场景来说，这意味着模型更擅长综合多段上下文，但也会受上下文长度和噪声影响。

### Q15-2: 预训练、SFT、RLHF 分别解决什么问题？
**高分回答：**  
- **预训练**：学语言和世界知识分布；  
- **SFT**：把“会说话”对齐成“按指令说话”；  
- **RLHF/DPO**：进一步对齐偏好、减少有害或低质量输出。  
落到工程里：我不能只依赖“模型很强”，还要通过流程约束和人审节点控制风险。

### Q15-3: 为什么模型会幻觉？你怎么在系统层面缓解？
**高分回答：**  
幻觉常见原因是参数记忆和上下文信号冲突、检索证据不足、或目标函数更偏“流畅”而非“真实”。  
我在系统层做三件事：先检索后回答、关键阶段人工确认、结果输出带来源与结构化约束。  
也就是说，不把“真实性”完全托付给单次生成。

### Q15-4: 温度（temperature）和 top-p 在生产里怎么用？
**高分回答：**  
温度控制随机性，top-p 控制采样尾部范围。  
生产任务（报告、抽取、规划）我倾向低温或接近贪心，保证稳定可复现；创意任务才提高随机性。  
在这个项目里，尽调报告属于高准确/高一致性场景，低温更合适。

### Q15-5: Prompt 工程和微调怎么取舍？
**高分回答：**  
优先级一般是：`prompt + 检索 + 流程约束` > 轻量微调 > 大规模微调。  
因为前者迭代快、成本低、可解释性好。  
只有当任务模式固定、数据量充足、线上收益明确时，我才会推动微调。

### Q15-6: RAG 的核心价值是什么？它的失败点在哪？
**高分回答：**  
价值是把“参数知识”扩展为“可更新外部知识”，提升时效性与可追溯性。  
失败点通常在检索召回不准、切片粒度不合理、重排弱、上下文拼接噪声大。  
所以 RAG 不是“加个向量库就行”，而是召回、重排、压缩、生成一整条链路优化。

### Q15-7: 你为什么用 LangGraph 编排，不直接写 while 循环调 LLM？
**高分回答：**  
while 循环适合短 demo，但在多阶段流程里会出现状态分散、恢复困难、分支爆炸的问题。  
LangGraph 把节点、路由、并行和中断变成显式结构，天然适合“可观测 + 可恢复 + 可扩展”的 Agent 系统。  
我的项目有 human interrupt 和多轮反馈，这类需求用图编排更稳。

### Q15-8: 为什么是“先 human_feedback 再 interview”，反过来行不行？
**高分回答：**  
可以反过来，但代价更高。先访谈再人审会把大量 token 花在可能错误的 analyst 方向上。  
先做 analyst 方案确认，相当于在低成本阶段做方向校准，能显著减少后续返工。  
这是典型的“把人工决策放在最能降成本的节点”。

### Q15-9: 为什么并行 interview，不串行 interview？
**高分回答：**  
并行的收益是缩短 wall-clock 时间，并提升维度覆盖。  
串行更容易做“前一步影响后一步”的深推理，但吞吐和等待体验会差。  
在尽调场景里，多维信息先并行采集再统一汇总，是更平衡的方案。

### Q15-10: 为什么要主图/子图分层，不做一个大图？
**高分回答：**  
一个大图在初期简单，但后期会出现节点职责混乱和复用困难。  
主图管流程编排，子图管单角色执行，是“控制平面/执行平面”分离思想。  
好处是复用性高，后续换工具或换角色只改子图，不动主干流程。

### Q15-11: 为什么你选择“任务状态机 + 后台线程”，不是消息队列？
**高分回答：**  
当前阶段目标是快速验证业务闭环，所以先用轻量运行时降低复杂度。  
但我也清楚边界：线程模型在多实例和高并发下不够稳。  
如果上线，我会迁移到队列（如 Celery/RQ）+ 外部存储，状态机语义保持不变，属于基础设施替换而非推倒重来。

### Q15-12: 你如何解释“为什么不是全自动 Agent”？
**高分回答：**  
高价值任务里，“可控”比“全自动”更重要。  
全自动短期看起来炫，但错误成本高、责任边界模糊。  
我采用的是“关键决策点人工把关 + 非关键环节自动化”，这是更工程化、更可落地的策略。

### Q15-13: 面试官追问“为什么不用 function calling 全部解决”怎么答？
**高分回答：**  
Function calling 是工具调用协议，不是完整流程系统。  
它解决“怎么调用工具”，但不直接解决“何时中断、如何重试、如何恢复、如何评估”。  
所以我把 function/tool 调用放在子图节点里，而流程级可靠性由编排层和运行时层承担。

### Q15-14: “为什么不用 MCP/为什么不用某个 Agent 框架”怎么答？
**高分回答：**  
框架选择标准应是需求匹配，不是追新。  
我的需求核心是状态图编排、中断恢复、多分支聚合，LangGraph 在这几点契合度高。  
如果后续外部工具生态和协议互通成为瓶颈，我会引入 MCP 作为工具层标准，而不是重写流程层。

### Q15-15: 如何回答“你的编排是不是过度设计”？
**高分回答：**  
我会看复杂度阈值：如果只是单轮问答，确实不需要图编排。  
但这个项目包含多阶段、可回退、人工介入和导出交付，隐含状态很多。  
用显式编排是把隐式复杂度前置管理，不是为了炫技。
