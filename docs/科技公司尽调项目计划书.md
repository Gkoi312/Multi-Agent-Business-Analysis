# 科技公司尽调改造计划书（贴合当前代码版）

## 1. 文档目的

本计划书用于把当前“通用研究报告系统”改造成“科技公司尽调系统”，并明确：

- 当前项目已经具备什么能力
- 本轮改造只做什么，不做什么
- 每一周落到哪些文件和接口
- 验收时如何判断是否做成

---

## 2. 当前系统现状（基于现有代码）

### 2.1 已具备能力（可直接复用）

- Web 流程已跑通：登录、Dashboard、任务列表、进度页
- 异步任务运行时已具备：任务创建、状态流转、失败重试、事件记录
- 多智能体研究链路已具备：主题输入 -> 生成分析员 -> 人工反馈 -> 继续生成报告
- 输出能力已具备：DOCX/PDF 下载
- 结构化日志已具备：JSON 日志（便于后续做可观测）

### 2.2 当前限制（必须承认）

- 输入字段只有 `topic`，没有“公司名/岗位/关注维度”结构化输入
- 报告模板偏“通用研究”，不是“尽调结构”
- 任务元信息里没有尽调相关字段（如风险等级、规模结论）
- 前端没有“证据卡片/风险表”展示，只有报告生成与下载

---

## 3. 本轮改造目标（作品集 MVP）

在不改动核心架构（FastAPI + TaskRuntime + LangGraph）前提下，实现一个可演示、可讲述的尽调版本：

1. 输入目标公司，产出结构化尽调报告  
2. 报告至少覆盖：业务、规模与发展、风险点  
3. 保留现有“任务追踪 + 失败重试 + 下载报告”能力  
4. 全量去除对外 `topic` 概念，改为尽调语义字段（作品集场景，无需兼容旧接口）

---

## 4. 业务输出定义（尽调报告模板）

最终报告固定为以下章节（先统一结构，再谈深度）：

1. 公司概览（做什么、核心产品、目标客户）
2. 业务拆解（收入/价值链/竞争位置）
3. 规模与发展（组织、融资、增长信号）
4. 风险点（市场、技术、合规、执行）
5. 结论（一句话判断 + 建议关注点）
6. Sources（保留引用）

说明：本轮先保证“有结构、有依据”，不强行承诺“财务精确值”。

---

## 5. 改造方案（按文件落地）

### 5.1 Web 层改造

#### 文件：`research_and_analyst/api/templates/dashboard.html`

- 页面文案改为“科技公司尽调助手”
- 表单字段统一为：
  - `company_name`（必填）
  - `focus`（可选，如“业务模式/风险/增长”）
  - `target_role`（可选，如“产品经理/算法工程师”）
- 删除输入侧 `topic` 字段，不再向用户暴露通用研究语义

#### 文件：`research_and_analyst/api/routes/report_routes.py`

- `POST /generate_report` 改为接收新字段
- 后端把结构化输入拼接为内部 `research_query` 传入服务层
- `task_type` 统一为 `company_due_diligence`
- `submit_feedback` 和 `report_progress` 模板参数同步去 `topic`，改为 `company_name`

### 5.2 任务层改造

#### 文件：`research_and_analyst/api/services/task_runtime.py`

- 在 `create_task` 中新增并持久化尽调元数据字段：
  - `company_name`
  - `focus`
  - `target_role`
  - `report_kind`（值：`due_diligence`）
- `topic` 字段从任务数据结构中移除（作品集模式不做向后兼容）
- 不改变现有任务状态机，继续沿用：
  - `running_generation -> awaiting_feedback -> running_feedback -> completed`

#### 文件：`research_and_analyst/api/services/report_service.py`

- `start_report_generation` 参数从 `topic` 更名为 `research_query`
- `get_report_status` 内文件命名优先使用 `company_name`（避免输出目录仍是 topic 语义）

### 5.3 Prompt 层改造

#### 文件：`research_and_analyst/prompt_lib/prompt_locator.py`

- 重点改造以下模板为“尽调导向”：
  - `CREATE_ANALYSTS_PROMPT`
  - `WRITE_SECTION`
  - `REPORT_WRITER_INSTRUCTIONS`
- 强制输出要求：
  - 必须覆盖业务、规模、风险三个维度
  - 每个风险点给出原因和影响
  - 保留来源引用格式

### 5.4 展示层改造（低成本）

#### 文件：`research_and_analyst/api/templates/tasks.html`

- 列表中 `Topic` 文案调整为“Company/Topic”
- 如果任务包含 `company_name`，优先显示公司名
- 保留事件查看和 Retry，不新增复杂前端组件

---

## 6. 分阶段计划（作品集冲刺版）

### 第 1 ：最小可用改造

- 完成 Dashboard 字段重构（`company_name/focus/target_role`）
- 完成 `generate_report` 与 `submit_feedback` 的参数替换（去 `topic`）
- 完成 Prompt 改造（尽调结构）
- 跑通单公司全流程：创建任务 -> 反馈 -> 生成 DOCX/PDF

交付物：

- 能输入公司名触发尽调
- 生成报告章节包含“业务/规模/风险”

### 第 2 ：稳定性与演示质量

- 在任务中持久化 `company_name/focus/target_role`
- 调整任务列表、进度页展示字段（全部去 topic 文案）
- 增加 5-10 条测试样例（不同公司名）做回归
- 固化 Demo 脚本（面试演示路径）

交付物：

- 连续 5 次任务执行可完成（允许少量内容差异）
- 页面和报告文案不再出现“通用研究”表述

---

## 7. 验收标准（务实）

满足以下条件即视为本轮完成：

- 用户输入 `company_name` 后，能完整走通现有任务链路
- 最终报告包含“业务、规模与发展、风险点”三块内容
- 任务列表可区分尽调任务（通过 `company_name` 或 `report_kind`）
- 失败任务可通过现有 Retry 接口重试成功
- 页面与接口对外不再出现 `topic` 字段

不作为本轮硬性指标：

- 精准财务建模
- 自动风险打分算法
- 复杂可视化面板

---

## 8. 风险与应对

### 风险 1：网络检索结果不稳定，报告质量波动

应对：在 Prompt 中固定报告结构，降低输出随机性；Demo 选取信息公开度高的公司。

### 风险 2：全量替换字段导致旧调用不可用

应对：本项目定位作品集，接受不兼容；通过一次性重构模板、路由、任务结构来降低长期维护复杂度。

### 风险 3：时间不足导致“改太大”

应对：优先改文案、输入、Prompt，不改底层工作流和数据库结构。
